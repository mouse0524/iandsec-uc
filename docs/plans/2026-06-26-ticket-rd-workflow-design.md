# 工单产研流转最小化设计

## 业务结论

本设计用于替换现有 Redmine 协作入口，把产品、研发、测试迁入当前系统，同时保持现有工单主流程稳定。

第一版不做完整项目管理系统，只做轻量产研任务池。工单继续承接用户、渠道、客服、现场闭环；产研任务承接产品评估、研发处理、测试验证和发布结论。

## 设计目标

- 保留现有工单状态，不重做用户侧流程。
- 用内部产研任务替换 Redmine issue。
- 支持一个工单关联多个产研任务，也支持多个工单关联同一个产研任务。
- 让产品、研发、测试在同一任务池中处理自己的待办。
- 产研完成后回到工单，由现场或提交方确认后关闭。

## 角色边界

用户负责提交问题、补充材料、现场验证、确认关闭和退回技术。

渠道负责代客户提交、补充现场信息、跟进客户确认和推动重开。第一版可沿用现有提交人和代理商权限。

客服负责受理、分拣、标准化描述、驳回无效工单和指派技术。

技术负责现场问题定位，是连接现场与产研的主负责人。技术判断是否转产研、创建或关联产研任务，并在产研完成后推动工单进入现场验证或待关闭。

产品负责需求确认、优先级、版本归属和需求驳回结论。

研发负责缺陷修复、需求开发、技术任务处理和提交测试。

测试负责复现、验证、退回研发和确认测试通过。

## 单据关系

保留现有 `Ticket` 作为外层工单。

新增 `R&D Task` 作为内部产研任务。一个工单可以关联多个产研任务，一个产研任务也可以关联多个工单。

典型场景：

- 多个客户反馈同一个缺陷时，多个工单关联到同一个缺陷任务。
- 一个客户问题拆成前端、后端、部署多个处理项时，一个工单关联多个产研任务。
- 一个产品建议沉淀为需求后，后续同类工单都关联到同一个需求任务。

## 工单主流程

工单继续使用现有状态：

- 待客服审核
- 客服驳回
- 技术处理中
- 现场验证
- 待关闭
- 技术驳回
- 已关闭

工单状态只表达客户侧闭环，不表达产研内部排期。

当工单需要产研处理时，仍停留在技术处理中。技术通过产研任务跟踪产品、研发、测试进度。

产研任务完成后，技术根据实际情况把工单流转到现场验证或待关闭。

## 产研任务状态

产研任务第一版使用以下状态：

- 待产品确认：需求类默认进入，产品判断是否采纳。
- 待研发处理：缺陷或已确认需求进入研发队列。
- 研发处理中：研发已接单处理。
- 待测试验证：研发提交测试。
- 测试不通过：测试退回研发。
- 测试通过：内部验证完成。
- 待发布：需要版本发布后才能现场确认。
- 已完成：已发布、已上线或无需发布即可交付结论。
- 已驳回：产品、研发或测试明确不处理。

状态流转按任务类型区分：

缺陷：待研发处理 -> 研发处理中 -> 待测试验证 -> 测试通过 -> 待发布/已完成。

需求：待产品确认 -> 待研发处理 -> 研发处理中 -> 待测试验证 -> 测试通过 -> 待发布/已完成。

技术任务：待研发处理 -> 研发处理中 -> 已完成。

## 产研任务最小字段

第一版字段控制在 Redmine 替代所需范围：

- 任务编号
- 标题
- 类型：缺陷、需求、技术任务
- 状态
- 优先级
- 产品负责人
- 研发负责人
- 测试负责人
- 计划版本
- 描述
- 处理结论
- 关联工单列表
- 创建人
- 操作日志

暂不引入迭代、Story Point、工时、甘特图、复杂看板和跨项目统计。

## 关键动作

工单侧新增两个动作：

- 转产研：从工单创建产研任务，并自动建立关联。
- 关联产研任务：把当前工单挂到已有产研任务。

工单侧替换原 Redmine 动作：

- 同步 Redmine -> 转产研任务
- 更新 Redmine -> 更新关联产研任务
- 拉取 Redmine -> 查看内部产研任务状态
- 打开 Redmine -> 查看产研任务详情

产研侧新增动作：

- 产品确认
- 产品驳回
- 指派研发
- 研发接单
- 提交测试
- 测试通过
- 测试退回
- 标记待发布
- 标记已完成
- 标记已驳回

所有动作必须写入产研任务日志。工单与产研任务的关联、取消关联和状态同步提示也必须留痕。

## 页面边界

第一版只做四类入口：

1. 工单技术处理页

   增加转产研、关联产研任务、查看关联任务。移除新流程中的 Redmine 主操作入口，历史 Redmine 信息保留只读展示。

2. 产研任务列表页

   产品、研发、测试共用一个任务池，通过任务类型、状态、负责人、优先级、计划版本筛选自己的待办。

3. 产研任务详情页

   展示任务信息、关联工单、负责人、状态流转、处理结论和操作日志。

4. 工单详情页

   展示关联产研任务、任务状态、负责人和最近处理结论。用户侧只看到必要结论，不暴露不必要的内部讨论。

## 权限边界

客服可以查看工单及其关联产研任务摘要，但不直接推进产研状态。

技术可以从自己可处理的工单转产研、关联产研任务，并查看关联任务详情。

产品可以处理待产品确认的需求类任务，并维护产品负责人、优先级和计划版本。

研发可以处理指派给自己的研发任务，提交测试或标记处理完成。

测试可以处理待测试验证任务，执行通过或退回。

管理员拥有全部查看和维护权限。

## 通知边界

第一版沿用现有工单提醒，不把产品、研发、测试提醒强塞进工单提醒配置。

产研任务内部提醒独立设计，最小规则：

- 转产研后提醒对应产品或研发负责人。
- 研发提交测试后提醒测试负责人。
- 测试退回后提醒研发负责人。
- 任务已完成或已驳回后提醒关联工单的技术负责人。

## Redmine 迁移边界

保留现有 Redmine 字段用于历史工单只读展示，不作为新流程入口。

新建工单默认不再要求同步 Redmine。

历史已关联 Redmine 的工单可以继续展示 Redmine 编号、链接和状态，后续是否迁移为内部产研任务另行处理。

第一版不做 Redmine 全量数据导入，不做双写，不做自动反向同步。

## 验证标准

设计落地后的最小验收标准：

- 技术可以从工单创建缺陷、需求或技术任务。
- 技术可以把多个工单关联到同一个产研任务。
- 产品可以确认或驳回需求任务。
- 研发可以接单、提交测试。
- 测试可以通过或退回。
- 产研任务完成后，关联工单不会自动关闭，只提示技术进入现场验证或待关闭。
- 工单详情能看到关联产研任务状态。
- 历史 Redmine 信息仍可只读查看。

## 第一版不做

- 不做完整项目管理系统。
- 不做迭代燃尽、工时统计、Story Point。
- 不做甘特图。
- 不做 Redmine 全量迁移。
- 不做产品、研发、测试三个独立工作台。
- 不改变现有工单主状态。
